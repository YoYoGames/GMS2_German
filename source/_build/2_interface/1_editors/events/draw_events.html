Ereignisse zeichnen

Das Draw-Ereignis bestimmt das, was Sie auf dem Bildschirm sehen, wenn Sie Ihr Spiel ausführen, und es ist in verschiedene separate Unterereignisse unterteilt, um die unterschiedlichen Zeichnungsanforderungen für Ihr Spiel besser zu bewältigen. Das Bild unten zeigt die Reihenfolge, in der die einzelnen Ereignisse ausgeführt werden: Das normale Draw- Ereignis kann in drei Unterereignistypen unterteilt werden: Der Draw-Anfang, der Standard- Draw und der Draw-End- Event. Im Allgemeinen müssen Sie nur das Standard- Unterereignis "Draw" verwenden, und Sie sollten wissen, dass bei allen Instanzen in einem Raum dieses Ereignis jeden Schritt des Spiels ausgelöst hat. Daher wird dieses Unterereignis immer aufgerufen, solange das Ereignis "." Bei der Instanz ist das sichtbare Flag auf true gesetzt. Selbst wenn Sie in den Objekteigenschaften nichts für dieses Unterereignis definiert haben (dh: kein Code oder keine Aktionen), wird dem Objekt, dem ein Sprite zugewiesen ist, immer noch ein Unterereignis zum Ziehen zugeordnet, das ausgelöst wird. Dies liegt daran, dass GameMaker Studio 2 zwei Arten im Draw- GameMaker Studio 2 zeichnen kann: Anwendung

  • Dies ist die Standardeinstellung, wenn Sie das Sprite in den Objekteigenschaften definieren und keine Aktionen oder Code in das normale Unterereignis Draw einfügen. In diesem Fall zeichnet GameMaker Studio 2 dieses Sprite. Beachten Sie, dass alle Transformationen, die Sie in anderen Ereignissen durchführen, um den Bildmaßstab, den Index, das Mischen usw. zu ändern, auch so lange angezeigt werden, wie das Zeichenereignis leer ist.

  • die benutzerdefinierte Ziehung, wenn Sie Code oder Aktionen in das Ziehereignis einfügen. Dies sagt GameMaker Studio 2 "Ich möchte steuern, was Sie für Instanzen dieses Objekts zeichnen" und überschreibt die Standardzeichnung vollständig. Dies bedeutet, dass Sie beispielsweise ein Objekt mit einem Sprite zugewiesen haben können. Dann können Sie das Zeichenereignis so einstellen, dass Text gezeichnet wird. Das Sprite wird nicht gezeichnet, da Sie GameMaker Studio 2 nicht angewiesen haben, es zusammen mit dem Text zu zeichnen.

Der Rest der Auslosungen wird in den folgenden Abschnitten erläutert:

Neben dem Hauptereignis Draw verfügen Sie auch über ein Unterereignis Draw Draw und End Draw. Diese verhalten sich genau wie ein standardmäßiges Draw -Unterereignis (dh sie zeichnen, was Sie in jedem Schritt des Spiels und für alle Ansichten einfügen), aber sie "zeichnen" nichts, wenn sie nicht zu einem Objekt hinzugefügt werden. Sie werden immer vor / nach dem Standard- Unterereignis Draw ausgeführt. Also, würden Sie die Draw Schritt für alle Instanzen beginnen, dann die Auslosung für alle Instanzen, dann schließlich das Draw End Schritt für alle Instanzen.

Auf diese Weise können Sie im Schritt "Zeichnungsbeginn" in einer Instanz Zeichen zeichnen oder Zeichnungseigenschaften festlegen und sicherstellen, dass alle Instanzen mit einem Standardereignis "Draw" oder "Zeichnungsende" diese Eigenschaften verwenden oder über das zeichnen, was die erste Instanz gezeichnet hat. Im Grunde ist es ein sicherer Weg, um sicherzustellen, dass bestimmte Dinge zu bestimmten Zeiten gezeichnet werden, ähnlich wie bei den Ereignissen Begin Step und End Step.

Beim Zeichnen von Dingen in GameMaker Studio 2 gibt es ein paar Dinge zu beachten, ob Sie ein Sprite, ein Shader oder 3D zeichnen:

  • Das Auslosungsereignis ist ein sehr intensives Ereignis, da es eines der Ereignisse ist, das die meiste Zeit und Ressourcen in Anspruch nimmt. Zu diesem Zweck ist es nie eine gute Idee, etwas anderes als das Auslosungsereignis zu tun. Speichern Sie also Ihren umfangreichen Code oder komplexe Aktionen für das Step-Ereignis oder Alarme oder was auch immer für ein anderes Ereignis passt. Lassen Sie das Zeichnungsereignis jedoch für die Zeichnung frei, da dies am besten ist.

  • Wenn der sichtbare Parameter für Ihr Objekt deaktiviert wurde (sodass die Instanzen des Objekts nicht gezeichnet werden), werden alle Zeichnungsereignisse übersprungen (mit Ausnahme des Resize- Ereignisses). Wenn Sie also eine Instanz unsichtbar machen, wird der gesamte Code, der in einem dieser Ereignisse platziert wurde, nicht ausgeführt. Fügen Sie also keinen wesentlichen Code in die Draw-Ereignisse ein, wenn sie nicht sichtbar sind.

  • Was Sie zeichnen, hat nichts mit der von Ihnen gewählten Kollisions-Engine (traditionell oder physikalisch) zu tun, da dies durch die Objekteigenschaften und das Sprite (oder die Maske) definiert wird, das das Objekt erhalten hat.

Der Zeichnungs-GUI - Unterereignistyp gehört zur Kategorie "Ereignis zeichnen " und ist speziell für das Zeichnen von GUI-Elementen (Graphical User Interface) konzipiert, die nicht von der Ansichtsskalierung oder -rotation beeinflusst werden. Dies bedeutet, dass Sie eine Instanz alle Elemente für Ihr HUD zeichnen lassen können, ohne die gesamte Positionierung der Elemente an der Instanzposition innerhalb des Raums oder an der Position der aktuellen Ansicht vornehmen zu müssen.

Wenn Sie in diesen Fällen zeichnen, ist es wichtig, dass Sie wissen, dass sich die Zeichnungskoordinaten nicht ändern, auch wenn Ansichten aktiv sind und (0,0) immer die obere linke Ecke der Anwendungsoberfläche oder der Anzeige ist (siehe Hinweis am unten in diesem Abschnitt), und die Standardbreite und -höhe beträgt 1: 1 mit der Anwendungsoberfläche. Die Tiefenreihenfolge wird immer noch zwischen verschiedenen Instanzen auf verschiedenen Ebenen beibehalten (dh eine Instanz auf einer niedrigeren Ebene wird unter einer höheren Ebene gezeichnet) und auch innerhalb der Ereignisse selbst, da das Unterereignis GUI Begin zeichnen zuerst für alle Instanzen zeichnet. dann wird die Standard- Draw-GUI für alle Instanzen darüber gezeichnet, und schließlich wird das Unterereignis Draw Gui End ausgelöst.

Auf diese Weise können Sie in der Draw GUI in einer einzigen Instanz Zeichnen oder Zeichnen von Eigenschaften festlegen. Stellen Sie sicher, dass alle Instanzen mit einer standardmäßigen Draw-GUI oder Draw-GUI -Endereignissen diese Eigenschaften verwenden oder über die erste Instanz zeichnen gezeichnet. Im Grunde ist es ein sicherer Weg, um sicherzustellen, dass bestimmte Dinge zu bestimmten Zeiten gezeichnet werden, ähnlich wie bei den Unterereignissen "Begin Step" und " End Step".

Diese Ereignisse können auch in Verbindung mit den normalen Draw -Unterereignissen verwendet werden (die normalerweise durch die Ansichtsposition, den Maßstab und die Drehung beeinflusst werden). Wenn Sie kein Draw-Ereignis, aber ein Draw-GUI-Event haben, zeichnet GameMaker Studio 2 das Sprite für die Instanz (sofern vorhanden) standardmäßig.

In Bezug auf die anderen Zeichnungs-Unterereignisse zeichnen die Draw-GUI- Ereignisse immer alles, was in den normalen Zeichnungsereignissen gezeichnet wird. Wenn Sie also eine Instanz auf einer niedrigeren Ebene mit einem Draw GUI- Ereignis haben, wird die Instanz, die sich auf einer höheren Ebene befindet, mit einem regulären Draw-Ereignis gezeichnet. Wenn beide Instanzen ein Draw-GUI -Unterereignis aufweisen, wird die Layer-Reihenfolge berücksichtigt.

HINWEIS: Dieses Ereignis zeichnet standardmäßig 1: 1 mit der Größe der Anwendungsoberfläche, die normalerweise der Größe des Raums oder des Ansichtsfensters entspricht. Dies bedeutet, dass die GUI nicht über die schwarzen Balken des "Letterbox" des Spiels angezeigt wird, wenn Sie in den Spieloptionen die Option " Seitenverhältniskorrektur " aktiviert haben. Dieses Verhalten kann mit dem ausgeschaltet werden display_set_gui_maximise Sie können das GUI-Ereignis auch auf eine bestimmte Größe sperren, die dann mithilfe der Funktion automatisch an die Abmessungen der Anzeige oder der Anwendungsoberfläche angepasst wird display_set_gui_size.

Die Pre Draw und Post - Draw Subereignissen Teil der Draw Event - Kategorie. Im Gegensatz zu den anderen Draw-Ereignissen werden diese jedoch direkt in den Bildschirmpuffer gezeichnet. Dies ist die Größe des kombinierten Bildschirmbereichs für alle derzeit sichtbaren Ansichten oder die Fenstergröße, wenn nur eine oder keine Ansicht verwendet wird. Das Bild unten veranschaulicht dies: Wenn Sie also die Pre- oder Post- Draw-Ereignisse verwenden, zeichnen Sie auf das Vollbild-Renderziel, das die gleiche Größe wie das Fenster hat, in das alle Ansichten passen. Wenn Sie keine Ansichten haben, wird stattdessen die Größe des Fensters selbst festgelegt.

Das Pre-Draw -Unterereignis wird vor jedem anderen Draw -Ereignis ausgelöst. Hier können Sie Werte festlegen, Zeichnungseigenschaften festlegen und sogar zeichnen, ohne sich um die Ansichten oder die Größe der GUI-Ebene kümmern zu müssen (die GUI-Ebenengröße kann gleich sein.) als Bildschirmpuffer, aber möglicherweise nicht, da Sie die GUI-Auflösung im Code einstellen können.

Es ist erwähnenswert, dass dieses Ereignis vor dem Löschen des Bildschirmpuffers auftritt. Wenn Sie also das Löschen von Ansichten im Raumeditor nicht ausschalten, wird nichts im Pre Draw- Ereignis angezeigt, da die erste Ansicht dies löscht. Wenn Sie die Ansichten selbst sehen möchten oder Ansichten überhaupt nicht verwenden, sollten Sie auch die Hintergrundfarbe im Raum-Editor auf einen Alpha-Wert von 0 eingestellt haben.

HINWEIS: Wenn Sie diese Optionen ausschalten, werden beim Testen des Spiels möglicherweise unerwünschte Artefakte auf dem Bildschirm angezeigt (z. B. "Spuren" von Instanzen). Das liegt daran, dass Sie direkt über den vorherigen Frame des Zeichenpuffers zeichnen, ohne dass dieser gelöscht wird. Sie können jedoch verwenden draw_clear_alpha das selbst zu tun.

Das Unterereignis Post Draw wird nach den Standard- Draw -Ereignissen ausgelöst, jedoch vor den Draw-GUI-Ereignissen. Wie beim Pre-Draw-Ereignis basiert es auf der Größe des Bildschirmpuffers und wird vor den Draw-GUI-Ereignissen platziert, damit Sie Nachbearbeitungseffekte und andere Dinge auf Vollbildschirmbasis einfach und problemlos ausführen können, ohne dass dies beeinträchtigt wird HUD / GUI-Elemente, die Sie möglicherweise in Ihrem Spiel haben.

Dieser Ereignistyp fällt unter die Kategorie Ereignis zeichnen, und obwohl er eigentlich nichts zeichnet, reagiert er auf Änderungen in der Zeichenfläche - speziell, um auf die Änderung der Größe der UWP-Leinwand zu reagieren, wenn sich das Spielfenster befindet "schnappte".

Auf der UWP-Zielplattform ist es wichtig, dass Sie die Größe der Spielleinwand im "Einrasten" des Spielfensters ändern (z. B. wenn der Benutzer das Fenster zur Seite des Bildschirms zieht). Dieses Ereignis wird genau dies tun und wird jedes Mal ausgelöst, wenn die Zeichenfläche geändert wird. Hier können Sie benutzerdefinierten Code hinzufügen, der die Ansicht ändert oder HUD-Elemente nach Bedarf neu positioniert. Diese Funktion ist eine bequeme Möglichkeit, um zu vermeiden, dass Schrittereigniscode ständig auf diese Dinge überprüft wird.

HINWEIS: Sie können bei diesem Ereignis nicht zeichnen! Es wird durch das Ändern der Größe der Leinwand ausgelöst und dient nur zum Erfassen dieser Änderung. Jede Zeichnung, die ausgeführt werden soll, muss sich noch in einem der anderen Zeichnungsereignisse befinden.
