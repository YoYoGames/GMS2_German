Dieses Ereignis tritt auf, wenn eine Instanz des Objekts zum ersten Mal erstellt wird. Dies ist das allererste Ereignis innerhalb einer Instanz, die über den Raumeditor in einem Raum platziert wird, wenn ein Raum betreten wird. Dies bedeutet, dass dieses Ereignis der ideale Ort ist, um Variablen zu initialisieren, Timelines zu starten, Pfade zu setzen usw. und alles andere zu tun, das normalerweise nur einmal oder nur dann ausgeführt werden muss, wenn eine Instanz zum ersten Mal im Raum erstellt wird. Wenn Ihr Objekt keine Objektvariablen oder Instanzvariablen hinzugefügt entweder in der Objekt - Editor oder dem Zimmer Editor, dann werden diese Variablen zunächst initialisiert werden und die dann wird Event erstellen ausgeführt werden.

Denken Sie daran, dass Sie alles, was Sie im create-Ereignis eingerichtet haben, über den Instanzerstellungscode im Raum-Editor ändern können, da dieser direkt nach dem create-Ereignis für die Instanz ausgeführt wird.


Dieses Ereignis ist das Ereignis, das ausgeführt wird, wenn eine Instanz zerstört wird. Es wird oft übersehen, wenn man Objekten Verhaltensweisen hinzufügt, aber es kann sehr nützlich sein, zum Beispiel durch das Erzeugen von Explosions- oder Partikeleffekten, wenn ein Gegner getötet wird, oder um eine neue Instanz des Objekts in einem anderen Teil des Raumes zu erzeugen sogar für das Hinzufügen von Punkten zu einer Partitur.


Dieses Ereignis wird nach jedem Ereignis aufgerufen, das eine Instanz des Objekts aus dem Raum entfernt. Es wird also ausgelöst, wenn die Instanz zerstört wird, wenn der Raum endet oder wenn das Spiel endet, und es dazu dient, dynamische Ressourcen zu bereinigen, die Sie in Ihrem Spiel haben (wie Oberflächen, Daten) Strukturen usw.) oder um eine Aufgabe auszuführen, die einmal ausgeführt werden muss, wenn die Instanz entfernt wird. Beachten Sie, dass dieses Ereignis nur aufgerufen wird, nachdem alle anderen Ereignisse aufgerufen wurden. Es ist also immer das letzte Ereignis, das in einer Instanz ausgeführt wird, wenn es aus dem Raum entfernt oder das Zimmer / Spiel beendet wird.

Aber was ist ein Alarm? Nun, es ist ein spezielles Ereignis, das nichts tut, wenn der Alarm nicht zuvor eingestellt wurde, und dann wartet es, bis der Alarm auf 0 heruntergezählt hat, bevor die Aktionen oder der Code ausgeführt werden, die Sie hinzugefügt haben. Beachten Sie, dass ein Alarm auf 0 setzt nicht den Alarmcode ausführen, wie das Ereignis ausgelöst wird, aber der Alarm sofort auf -1, so dass der Code übersprungen. Wenn Sie einen Alarm benötigen, um den nächsten Schritt auszuführen, sollten Sie ihn auf 1 setzen.

Dies kann sehr nützlich sein, da es Ihnen ermöglicht, Dinge in bestimmten Momenten in Bewegung zu setzen, und Sie können sie sogar wiederholen lassen, da es nichts gibt, was Sie daran hindern könnte, einen Alarm für sich selbst zu setzen. Natürlich ist es sehr wichtig, dass Sie wissen, wann zwei (oder mehr) Instanzen eines Objekts kollidiert sind, und dafür haben wir das Kollisionsereignis. Dies ist ein Ereignis, das Sie in ein Objekt einfügen und dann angeben, für welches andere Objekt Sie nach Kollisionen suchen sollen.

Wenn Sie die Physik nicht aktiviert haben, werden diese Kollisionen basierend auf der Maske der beiden Objekte berechnet (die Maske wird in den Sprite-Eigenschaften definiert oder kann unabhängig in den Objekteigenschaften zugewiesen werden) und ob sie sich überschneiden oder nicht. Beachten Sie, dass, wenn der einen oder anderen Instanz in der Kollision keine Maske zugewiesen ist (oder die Sprite-Maske auf nichts gesetzt ist), selbst wenn sie gezeichnet wird, keine Kollisionen erkannt werden. Wenn Sie Physik aktiviert haben, basiert die Kollision auf der Art der Kollisionsform ( Fixture ), die Sie für das Objekt in seinen physikalischen Eigenschaften definiert haben, sowie auf der Kollision. Das bedeutet, dass Sie möglicherweise keinen Code benötigen, um die Kollision zu behandeln, aber dieses Ereignis muss noch mindestens einen Kommentar enthalten, damit die Kollisionen erkannt werden.

Ereignisauftrag

Bei der Betrachtung von Ereignissen in GameMaker Studio 2 ist zu beachten, dass die genaue Reihenfolge der Ereignisse in jedem Schritt nicht eindeutig festgelegt werden kann, da dies von den internen Funktionen von GameMaker Studio 2 abhängt und sich dies ändern kann Die Software entwickelt sich. Es gibt jedoch bestimmte Ereignisse, die immer in der gleichen Reihenfolge ausgeführt werden. Die erste Reihe von Ereignissen, die immer auf die gleiche Weise passieren, sind diejenigen, die auftreten, wenn ein Raum zum ersten Mal betreten wird. Die Reihenfolge, in der die verschiedenen Ereignisse ausgelöst werden, lautet:

  • Objektvariablen / Instanzvariablen werden initialisiert (dies sind die Variablen, die Sie in der IDE des Objekt-Editors oder des Raum-Editors definieren können ). Wenn Sie ein Objekt mit angehängten Variablen haben, werden diese vor allem anderen erstellt, und danach werden alle Instanzen mit angehängten Variablen definiert, sodass Sie Objektvariablen im Raumeditor mit instanzspezifischen Variablen überschreiben können.

  • Erstellen Sie das Ereignis, gefolgt vom Instanzerstellungscode jeder Instanz. Damit jede Instanz erstellt wird, wird zuerst ihr Create-Ereignis und dann der Instanzerstellungscode ausgeführt, bevor die nächste Instanz erstellt wird

  • Game Start Event - Dieses Event wird einmal im allerersten Raum des Spiels für alle Instanzen ausgelöst, die vom Raumeditor in den Raum platziert wurden (beachten Sie, dass Calling game_restart() wird dieses Ereignis erneut auslösen.

  • Raumerstellungscode - Dies ist der einmalige Code, der im Raumeditor für die erstmalige Eingabe eines Raums geschrieben wird

  • Room Start Event aller Instanzen - eine der "anderen" Kategorie der Ereignisse und wird für alle Instanzen ausgelöst, persistent oder anderweitig


Beachten Sie auch, dass Sie die Reihenfolge, in der bestimmte Instanzen erstellt werden, auch im Raumeditor selbst festlegen können, indem Sie sie in der Liste des Fensters Eigenschaften der Instanzebene nach oben oder unten verschieben.

Abgesehen von diesen spezifischen Ereignissen ist die einzige bekannte Reihenfolge immer gleich, unabhängig davon, was zu den drei Schritt- Ereignissen und den verschiedenen Zeichenereignissen gehört. Diese werden immer konsistent bleiben. Wenn Sie Code haben, der während jedes Spielschritts auf ein bestimmtes Timing angewiesen ist, sollten Sie Folgendes verwenden:

  • Beginnen Sie Schritt Ereignis

  • Schritt-Ereignis (Beachten Sie, dass das Schritt-Ereignis ausgeführt wird, kurz bevor Instanzen an ihren neuen Positionen platziert werden)

  • Ende Schritt Ereignis


Alle Unterereignisse für das Zeichnen werden auch immer in der folgenden Reihenfolge behandelt (außer beim Ereignis Resize, das anders ausgelöst wird):

  • Vorziehung Ereignis

  • Zeichnen Sie das Ereignis ein

  • Ereignis zeichnen

  • Ende Ereignis zeichnen

  • Nachzeichnen-Ereignis

  • Draw GUI Beginne Ereignis

  • Zeichnen Sie ein GUI-Ereignis

  • Draw GUI End Event
