Der Objekt-Editor


Objekte sind die Ressource, die wir verwenden, um Aspekte eines Spiels zu steuern und bestimmte Dinge zu tun. Die meiste Zeit haben sie ein Sprite mit ihnen verbunden, so dass Sie sie in den Spielräumen sehen, aber manchmal werden sie als "hinter den Kulissen" Controller verwendet, um Dinge in Bezug auf den Benutzer oder für das Timing usw. zu tun Sie können Verhaltensweisen erhalten und auf bestimmte Ereignisse sowie auf einander reagieren. Die meisten Dinge, die Sie in einem Spiel sehen, basieren auf Objekten und ihren Interaktionen. Jetzt sagen wir "basierend auf", weil Sie Objekte nicht direkt in die Spielräume platzieren, sondern stattdessen Instanzen dieser Objekte platzieren, die im Grunde Kopien (oder Klone, wenn Sie bevorzugen) der Objektressource. Dies ist eine sehr wichtige Sache, die man sich merken sollte, da Instanzen und Objekte nicht die gleiche Sache sind und jede ihre eigenen Funktionen hat, die sie beeinflussen können.

Um das zu verdeutlichen, denke an ein Spiel, das mit GameMaker Studio 2 erstellt wurde. Alle Charaktere, Monster, Bälle, Wände, usw. sind alle Objekte, die Sie im Ressourcenbaum erstellen. Dann ziehen Sie das Objekt in einen Raum im Raum-Editor, um eine Instanz zu erstellen, die aus der grundlegenden Objektvorlage kopiert wurde. Die Instanz in dem Raum kann dann im Editor selbst oder durch Code geändert werden, so dass sie skaliert oder gefärbt werden kann oder andere Details von dem "Basis" -Objekt modifiziert werden können. Wenn wir also über etwas sprechen, das eine Instanz beeinflusst oder verändert, meinen wir, dass es sich auf eine bestimmte Kopie eines Objekts in einem Raum auswirkt. Nur diese einzelne Instanz wird beeinflusst, während der Rest nicht betroffen ist. Aber wenn wir davon sprechen, ein Objekt zu beeinflussen oder zu ändern, meinen wir, dass wir das Objekt im Ressourcenbaum modifizieren und alles, was wir damit tun, wird in allen Instanzen widergespiegelt, die von diesem Punkt an erstellt werden. Das Objekt ist also die Vorlage für die Instanz und die Instanz ist, was wir in einem Raum platzieren, um unser Spiel zu erstellen.

Um ein Objekt in Ihrem Spiel zu erstellen, können Sie mit der rechten Maustaste klicken Klicken Sie im Objektressourcenordner auf die Option Erstellen. Dies öffnet das Fenster oben auf dieser Seite mit den folgenden Abschnitten:

Hier können Sie Ihrem Objekt einen Namen geben. Dieser Name darf niemals Leerzeichen oder nicht autorisierte Zeichen enthalten (nur alphanumerische Zeichen und die Unterstrichangabe "_" sind erlaubt) und sollte kurz, aber einprägsam sein, damit Sie Ressourcen später leichter identifizieren können, wenn Sie viele davon haben. Zum Beispiel verwenden viele Leute ein Präfix oder ein Suffix, um die Unterschiede zwischen Ressourcen wie "obj_Ship" oder "Ship_obj" zu identifizieren.

Als nächstes können Sie das Sprite für das Objekt auswählen, wenn Sie möchten, dass es ein Objekt hat. Klicken Sie mit der linken Maustaste In der Sprite-Box wird der Asset Explorer mit einer Liste aller verfügbaren Sprites aus dem Ressourcenbaum angezeigt:

Unterhalb des Sprite-Zuweisungsbereichs haben Sie eine Schaltfläche zum Auswählen einer bestimmten Kollisionsmaske. Standardmäßig verwendet ein Objekt die Kollisionseigenschaften des zugewiesenen Sprites. Es gibt jedoch Fälle, in denen Kollisionen auf einer anderen Maskenform basieren sollen. Zum Beispiel haben Sie vielleicht ein Top-Down-Spiel, bei dem das Hauptsprite eine rechteckige Kollisionsmaske hat, aber um eine Kollision glatter zu machen, kann eine kreisförmige Maske besser sein. In diesen Fällen können Sie auf die Schaltfläche Kollisionsmaske klicken und eine andere Sprite-Ressource auswählen, sodass Instanzen des Objekts jetzt das zugewiesene Sprite, aber Basiskollisionen vom Masken- Sprite zeichnen.

Sichtbar zeigt an, ob Instanzen dieses Objekts beim Start des Raums sichtbar sind. Normalerweise sind die meisten Instanzen sichtbar, aber manchmal ist es nützlich, unsichtbare zu haben - zum Beispiel können Sie sie als Wegpunkte verwenden, um ein bewegtes Monster zu steuern oder um Werte zu verfolgen und bestimmte zeitgesteuerte Aktionen auszuführen. Unsichtbare Objekte reagieren immer noch auf Ereignisse und wenn ihnen ein Sprite oder eine Maske zugewiesen wurde, reagieren sie auch, wenn andere Instanzen mit ihnen kollidieren... sie sind einfach nicht zu sehen und führen ihr Zeichenereignis nicht aus. Wenn Sie eine Instanz als solide GameMaker Studio 2, weisen Sie GameMaker Studio 2 dass es versuchen sollte, alle Kollisionen zu lösen, bevor ein Kollisionsereignis ausgelöst wird. Dies ist eine sehr grundlegende Funktionalität, und in Wirklichkeit wird die Instanz nur in die Position zurückversetzt, in der sie sich befand, bevor eine Kollision erkannt wurde, bevor Code oder DnD™ ausgeführt wurde, der in dem Ereignis selbst enthalten ist begrenzte Verwendung.


Persistent - Ein persistentes Objekt ist ein Objekt, das nicht verschwindet, wenn der Raum verändert wird, sondern "persistiert" und in den neuen Raum übertragen wird. Es wird nur verschwinden, wenn Sie es explizit zerstören, entweder mit einer Zerstörungsaktion oder im Code. Wenn Sie also ein persistentes Objekt in einem Raum platzieren, ist es in allen folgenden Räumen weiterhin verfügbar und führt weiterhin alle Aktionen aus, die den Ereignissen in seinen Objekteigenschaften zugewiesen sind, mit Ausnahme des Ereignisses create, das nur einmal ausgelöst wird wird zunächst erstellt und nicht, wenn Sie in ein neues Zimmer wechseln.

Dies ist praktisch, wenn Sie beispielsweise einen Hauptcharakter haben, der sich von Raum zu Raum bewegt, und Sie möchten die Variablen in diesem Objekt beibehalten, aber Sie sollten sorgfältig darauf achten, persistente Objekte korrekt zu erstellen und zu zerstören Fehler, die sich irrtümlich in Ihr Projekt eingeschlichen haben.

Beachten Sie auch, dass bei einem persistenten Objekt die Ereignisse Spielstart, Spielende, Raumstart und Raumende immer noch ausgelöst werden. Wenn Sie das Spiel jedoch neu starten (z. B. mit der Funktion game_restart ()), werden alle permanenten Objekte entfernt existieren, wenn sie vom Spiel neu erstellt werden. Beachten Sie außerdem, dass wenn Sie ein persistentes Objekt deaktivieren, es nicht mehr von einem Raum in einen anderen übergeht, es sei denn, es wird wieder aktiviert, bevor das Ereignis Raumende ausgelöst wird.


Wenn Sie das Kästchen Physik verwenden aktivieren, sagen Sie GameMaker Studio 2 dass dieses Objekt Teil einer physikalischen Welt sein soll. Wenn Sie dies überprüfen, wird auch ein weiteres verkettetes Fenster geöffnet, in dem Sie die physikalischen Eigenschaften aller Instanzen dieses Objekts definieren können. Mehr Informationen dazu finden Sie weiter unten auf dieser Seite im Bereich Physik.



Optionen

Der Abschnitt Optionen definiert die verschiedenen Eigenschaftsoptionen für Ihr Objekt und wie sich die Instanzen verhalten, wenn sie in einem Raum platziert werden. Hier fügen Sie Ereignisse hinzu, die die "Bausteine" Ihres Spielcodes sind, legen fest, ob das Objekt einen Elternteil haben soll, und legen auch die Physikeigenschaften fest, wenn Sie die Physikfunktionen aktivieren möchten.

Also, was sind Ereignisse? Im Grunde genommen sind dies diskrete Momente in der Spieleschleife, in denen Dinge basierend auf dem, was Sie für sie programmiert haben, geschehen. GameMaker Studio 2 arbeitet mit Zyklen dieser Ereignisse - von dem Moment, in dem ein Raum gestartet wird, bis zu dem Zeitpunkt, an dem es fertig ist, läuft ein Spiellauf, bei dem jeder Schritt (ein Schritt ist ein Moment der Spielzeit, abhängig von der Raumgeschwindigkeit) ist Eine Reihe von Ereignissen wird ausgeführt oder überprüft, und Sie können festlegen, dass Code- oder DnD-Aktionen in Ihren Objekten platziert werden, die auf diese Ereignisse reagieren.

So kann jedes Objekt in Ihrem Spielprojekt ein Elternobjekt haben, aber was bedeutet das? Nun, wenn einem Objekt ein Elternteil zugewiesen ist, kann es Code, Aktionen und Ereignisse mit diesem Elternteil teilen. Diese Freigabe wird "Vererbung" genannt, und ein Objekt, das einen Elternteil hat, wird als "Kind" -Objekt bezeichnet. Ein Kind kann nicht nur Code mit seinem Elternteil teilen, sondern Sie können auch Überprüfungen durchführen und Code für Elternobjekte ausführen und es wird automatisch auch die Kindobjekte einschließen, was viel Zeit und Energie spart.

Wenn das kompliziert klingt, ist eine andere Möglichkeit, ein Elternobjekt zu betrachten, eine Möglichkeit, Objekte unter demselben Schirm zu "gruppieren" und sie bestimmte Dinge teilen zu lassen, ohne ihre eigene einzigartige Identität zu verlieren. Vielleicht klärt das die Dinge noch nicht zu sehr auf, also lassen Sie uns ein paar Beispiele geben...

ABER wenn wir ein Elternobjekt für alle Feinde machen, können wir nur ein Kollisionsereignis mit dem Elternobjekt erzeugen und es wird ausgelöst, egal welches der vier "Kinder" -Feinkörperobjekte den Spieler berühren. Beachten Sie, dass ein übergeordnetes Objekt keine Ereignisse oder Code enthalten muss...

Ein anderes Beispiel für Elternschaft wäre, wenn Sie ein Spiel haben, in dem Sie 10 verschieden aussehende Objekte erstellen möchten und sich alle auf die gleiche Weise verhalten. Dazu würden Sie ein Elternobjekt erstellen, in dem alle Verhaltensaktionen oder Code in den erforderlichen Ereignissen enthalten wären, und dann würden Sie Ihre zehn Objekte ohne Aktionen oder Code, aber unterschiedliche Sprites erstellen und ihnen Ihr übergeordnetes Objekt zuweisen. Wenn Sie nun diese Instanzen in einem Raum platzieren, verhalten sie sich alle gleich, sehen aber anders aus, da sie die Ereignisse des Elternteils "erben".

Schließlich können Sie Eltern verwenden, um Ereignisse und Verhaltensweisen zu "mixen und anzupassen". Wir werden dies mit einem letzten Beispiel illustrieren... Sagen wir, du willst zwei Monster haben: eins, das sich auf und ab bewegt, während ein anderes sich nach links und rechts bewegt, aber du willst auch, dass die zwei die gleiche Gesundheit haben, schieße auf die Spieler und verletzt den Spieler, wenn sie mit ihnen kollidieren. In diesem Fall können Sie sehen, dass fast alle Ereignisse die gleichen Aktionen haben sollten, mit Ausnahme von ein oder zwei, die die Bewegung steuern. Also können wir wiederum ein Objekt zum übergeordneten Objekt machen, aber in diesem Fall definieren wir auch bestimmte Ereignisse für das untergeordnete Objekt. Diese Ereignisse "überschreiben" die übergeordneten Ereignisse, was bedeutet, dass immer dann, wenn ein Ereignis für das untergeordnete Objekt Aktionen enthält, diese anstelle der Aktionen ausgeführt werden, die im Ereignis des übergeordneten Objekts enthalten sind. Wenn Sie auch das übergeordnete Ereignis ausführen möchten, können Sie das so genannte "geerbte" Ereignis mit der Funktion aufrufen event_inherited() oder die DnD™ -Aktion "Parent Event" aufrufen.

Wenn Sie ein übergeordnetes Objekt im Code auswählen, wird der Code auch auf die "untergeordneten Elemente" des übergeordneten Objekts angewendet. Dies geschieht, wenn Sie in einer Aktion angeben, dass die Aktion auf Instanzen eines bestimmten Objekts angewendet werden muss, und im Code bei der Verwendung des with() Erklärung. Es funktioniert auch so, wenn Sie Code-Funktionen aufrufen, wie instance_position, instance_number usw. Wenn Sie ein übergeordnetes Objekt angeben, werden alle Instanzen der über- und untergeordneten Instanzen in die Überprüfungen eingeschlossen. Schließlich funktioniert Parenting auch dann, wenn Sie sich auf Variablen in anderen Objekten beziehen, wie im Beispiel oben, wenn ich die Geschwindigkeit des Feindes 1 auf 10 setze, dann wird die Geschwindigkeit des Gegners 2 ebenfalls auf 10 gesetzt, da es ein Kindobjekt von Feind 1 ist.

In den meisten Fällen wird es als gute Übung angesehen, ein Basis-Elternobjekt zu erstellen, und dieses Basisobjekt enthält das Standardverhalten, aber niemals eine Instanz davon im Spiel. Verwenden Sie stattdessen alle untergeordneten Objekte und verwenden Sie das Elternelement nur in Situationen wie den oben genannten, für Kollisionen, zum Referenzieren von Variablen usw. Sie sollten auch erkennen, dass Eltern auch Eltern haben können! Offensichtlich können Sie keinen Zyklus von "Eltern 1 ist Kind von Eltern 2 ist Kind von Eltern 1" erstellen, aber Sie können erstellen, was "Objekthierarchie" heißt, wobei "Eltern3 ist Kind von Eltern2 ist Kind von Eltern1". Dies ist äußerst nützlich, um Ihr Spiel strukturiert zu halten und Ihnen wird dringend empfohlen, diesen Mechanismus zu verwenden.

Die Dichte von etwas ist definiert als ihre Masse pro Einheitsvolumen, was im Grunde bedeutet, wie viel Masse in den Raum eingepfercht ist, den es in der Welt aufnimmt. So hätte ein Ballon eine geringe Dichte, da er eine geringe Masse hat, die einen großen Raum einnimmt, während ein Führungsstab eine hohe Dichte haben würde, da er eine große Masse hat und dennoch wenig Raum einnimmt. In der 2D- GameMaker Studio 2 wird die Masse automatisch aus den Werten berechnet, die Sie für die Dichte und die Fläche der Form eingeben, die Sie für das Fixture definieren. Dies hat einen direkten Einfluss darauf, wie viel Trägheit eine Leuchte hat und wie sie auf Kollisionen und Kräfte reagiert. Wenn man also eine kleine Form mit hoher Dichte macht, hat sie eine sehr große Masse (wie ein Bleistab), Wenn Sie jedoch eine große Form mit einer geringen Dichte definieren, hat sie eine viel kleinere Masse (wie ein Ballon).


In der Physik ist Restitution definiert als " die Rückkehr eines Objekts oder Systems in seinen ursprünglichen Zustand nach elastischer Verformung ", aber da die Halterungen im GameMaker Studio 2 wirklich starre Körper sind und nicht deformiert werden können, ist Restitution eine Art zu sagen wie "federnd" ist die Leuchte. Diese Einstellung beeinflusst, wie stark ein Objekt "springt", wenn es mit anderen Objekten kollidiert, und ist abhängig von anderen Kräften, die auf die Instanz wirken, wie beispielsweise Schwerkraft und Reibung.


Standardmäßig ist die Kollisionsgruppe für alle Physikobjekte 0 und dies bedeutet, dass sie wie gewohnt interagieren, was wiederum bedeutet, dass sie ein Kollisionsereignis haben müssen (ein einfacher Kommentar genügt, um die Kollision auszulösen), sonst wird keine Kollision stattfinden angemeldet sein. Sie können jedoch auch angeben, dass ein Objekt zu einer bestimmten Kollisionsgruppe gehören soll. Wenn Sie Objekte einer positiven Kollisionsgruppe zuweisen (dh drei Objekte befinden sich alle in Gruppe " 2 "), weisen Sie GameMaker Studio 2 an, dass Instanzen dieser Objekte immer kollidieren sollen, auch wenn sie in ihrem Kollisionsereignis nichts haben und den physikalischen Eigenschaften gehorchen dass Sie ihnen und dem Raum, in dem sie sich befinden, zugewiesen haben. Umgekehrt, wenn Sie Objekte mit einer negativen Kollisionsgruppe haben (dh vier Objekte mit " -1 "), weisen Sie GameMaker Studio 2 dass diese niemals kollidieren sollten Ereignisse zwischen ihnen werden ignoriert.

HINWEIS: Die Verwendung von Kollisionsgruppen erhöht die vom Physiksystem benötigte Verarbeitung dramatisch und sollte nur dann verwendet werden, wenn es absolut notwendig ist. Sie sollten so wenig Gruppen wie möglich haben.

Dämpfung wird verwendet, um die physikalische Weltgeschwindigkeit einer Instanz zu reduzieren, und unterscheidet sich von der Reibung darin, dass Reibung nur auftritt, wenn zwei Instanzen mit Kollisionsformen (Vorrichtungen) in Kontakt sind. Dämpfung ist auch viel billiger zu simulieren als Reibung, aber bitte beachten Sie, dass die Dämpfung kein Ersatz für Reibung ist; Die zwei Effekte können und sollten zusammen verwendet werden.


Wenn Sie an ein sich drehendes Objekt in der "realen Welt" denken, es sei denn, es hat einen Motor oder ist es im Raum, wird es im Laufe der Zeit aufgrund des Einflusses äußerer Kräfte (wie Reibung mit der umgebenden Luft) langsamer. Wir können diese Option verwenden, um diesen Effekt zu simulieren und die Rotationsgeschwindigkeit von Instanzen in der Physikwelt zu reduzieren, da ohne sie jede rotierende Instanz unendlich weiter rotieren würde.


Reibung ist die Kraft, die der relativen Bewegung von gegeneinander gleitenden Materialelementen widersteht, was sich in der GameMaker Studio 2 als Impulsverlust auswirkt, der durch die Kollision von zwei Instanzen mit an sie gebundenen Scheinwerfern verursacht wird. Wenn also zwei Instanzen kollidieren, wird ihre Bewegung von diesem Wert beeinflusst, wobei eine hohe Reibung einen größeren Impulsverlust verursacht als ein niedrigerer Wert.

Wenn Sie dieses Kontrollkästchen aktivieren, weisen Sie GameMaker Studio 2 an, die physischen Eigenschaften des Objekts grundsätzlich zu "ignorieren", aber dennoch ein Kollisionsereignis mit den umgebenden Objekten zurückzugeben. Auf diese Weise können Sie eine Instanz ohne physische Anwesenheit im Raum haben, die aber auf die Kollision einer anderen Instanz wie dem Spieler reagieren und wiederum etwas tun kann, zum Beispiel eine Tür öffnen oder eine Aktion in einer anderen auslösen Platz im Zimmer.

HINWEIS: Eine Sensorvorrichtung löst das Kollisionsereignis nur dann aus, wenn die Kollision zuerst auftritt. Dies bedeutet, dass Sie keinen Strom von Kollisionsereignissen erhalten, da sich die beiden Körper weiterhin überlappen (was normalerweise der Fall ist). Wenn sie sich nicht mehr überlappen und sich überlappen, wird ein weiteres Kollisionsereignis ausgelöst.

Dies ist standardmäßig aktiviert und stellt den anfänglichen "Zustand" des Scheinwerfers am Anfang des Raums dar, in dem er platziert wurde. Normalerweise sollte er von Anfang an wach sein (dh: Teilnahme an der Physikwelt), aber manchmal Dies kann aufgrund des abrupten Beginns der Physiksimulation zu bestimmten Instabilitäten und unerwünschten Effekten führen. Um diese Effekte zu vermeiden, können Sie dieses Flag deaktivieren, und die Instanz wird mit eingeschalteter Appliance erstellt, und sie wird nicht an der Physiksimulation teilnehmen, solange nicht etwas darauf wirkt.


Es wird bestimmte Fälle in einem Spiel auf Physikbasis geben, die du bewegen willst, aber nicht durch Kräfte wie Schwerkraft oder Kräfte, die durch Kollisionen mit dynamischen Objekten entstehen (z. B. bewegliche Plattformen in einem Plattformspiel) ). Für solche Objekte bedeutet das einfache Einstellen der Dichte der Vorrichtung auf 0, dass die Physik davon ausgeht, dass das Objekt statisch sein soll und überhaupt nicht auf irgendetwas reagieren wird. Wenn Sie dieses Kontrollkästchen aktivieren, wird ein statisches Objekt kinematisch, und obwohl es von Kollisionen und der Schwerkraft nicht beeinflusst wird, kann es immer noch mit den entsprechenden Variablen verschoben oder gedreht werden.

Die verschiedenen Arten von Variablen, die Sie erstellen können, sind unten aufgeführt:

  • Real: Eine reelle Zahl ist eine beliebige Zahl, die keine ganze Zahl ist und positiv oder negativ sein kann. Also, 124,5, 45639,566546456, 0,9, -45,5 etc... sind alle Beispiele für reelle Zahlen. Alle reellen Zahlen werden als 32-Bit-Gleitkommawerte gespeichert, so dass es bei diesen Problemen zu leichten Rundungsfehlern kommen kann. Beachten Sie, dass Sie in den Variablenoptionen einen Bereich von Werten für die Ausgabe reell festlegen können (siehe unten für weitere Details).

  • Ganzzahl: Eine Ganzzahl ist eine ganze Zahl und kann positiv oder negativ sein, z. B. 30004, 19, 0, -300. Beachten Sie, dass Sie einen Wertebereich für die Ausgabe-Ganzzahl aus den Variablenoptionen festlegen können (weitere Einzelheiten finden Sie weiter unten).

  • String: Ein String ist alles, was in doppelte Anführungszeichen gesetzt wurde, zum Beispiel "Fisch", "Hallo Welt" oder "12345".

  • Boolean: Ein boolescher Wert ist entweder ein Wert true oder false. Im Fenster Objekt-Editor-Variablen wird dies einfach als ein Feld angezeigt, nach dem Sie suchen true und deaktivieren Sie für false.

  • Ausdruck: Ein Ausdruck ist eine mathematische Phrase, die gewöhnliche Zahlen, Variablen, Strings oder GML-Funktionen sowie einen oder mehrere Operatoren enthalten kann. Beispielsweise sqrt(85 * 6) + 5.5 ist ein Ausdruck.

  • Ressource: Eine Ressource ist einfach eine der Ressourcen, die Sie im Ressourcenbaum definiert haben. Wenn Sie diesen Ressourcentyp auswählen, können Sie auf den Open Asset Explorer klicken Klicken Sie auf, um den Asset Explorer zu öffnen und die gewünschte Ressource auszuwählen. Beachten Sie, dass Sie Filter für die Assets, die im Asset Explorer angezeigt werden, über die Variablenoptionen festlegen können (siehe unten für weitere Details).

  • Liste: Wenn Sie einen Listentyp für die Eingabe auswählen, können Sie eine Auswahl von Werten erstellen (diese können Strings, Reals, Ausdrücke usw. sein), von denen Sie dann einen oder mehrere auswählen können, damit die Variable zurückgegeben wird. Um die Elemente zu definieren, die in die Liste aufgenommen werden, müssen Sie sie zuerst den Variablenoptionen hinzufügen (siehe unten für weitere Details) und dann können Sie das Element als Standardwert für die Variable im Hauptfenster aus dem Drop auswählen Menü unten: Beachten Sie, dass bei Auswahl von Mehrfachauswahl in den Listenoptionen (siehe unten) die Variable zu einem Array aller ausgewählten Optionen wird.

  • Farbe: Mit dem Farbtyp definieren Sie einen Farbwert, der in der Variablen gespeichert werden soll. Sie können eine reelle Zahl (von 0 bis 16777216, diese hat einen Alpha-Wert von 255) oder einen Hex-Wert (im Format von $ RRGGBBAA) eingeben, oder Sie können einen Doppelklick ausführen auf dem Farbfeld, um die Farbauswahl zu öffnen und die Farbe dort definieren.
