Kollisionsprüfung

Bei der Planung von Bewegungen oder der Entscheidung für bestimmte Aktionen ist es oft wichtig zu sehen, ob es Kollisionen mit anderen Objekten an bestimmten Orten in der Spielwelt gibt, und oft ist es die wichtigste Aufgabe, die richtige Kollision für den Job zu wählen. GameMaker Studio 2 verfügt über eine Reihe von integrierten Funktionen, mit denen Sie Kollisionen korrekt und in Übereinstimmung mit den Anforderungen Ihres Projekts behandeln können.

WICHTIG! Alle Kollisionsfunktionen arbeiten unabhängig von den Eingabewerten mit ganzzahligen Positionen. Alle nicht ganzzahligen Werte werden auf die nächste Ganzzahl gerundet.


Die folgenden Funktionen beschäftigen sich mit den verschiedenen Möglichkeiten, wie Sie eine Instanz auf Kollisionen mit einer anderen Instanz prüfen lassen können:

  • Ort_leer
  • Platz_frei
  • place_meeting
  • position_empty
  • position_treffen
  • Positionsänderung
  • position_destroy
  • instance_place
  • Instanzposition

Die folgenden Routinen können dafür verwendet werden (sowie andere Funktionen, die in den Abschnitten über das Bewegen und Instanzen beschrieben werden ). Diese Kollisionsfunktionen funktionieren alle etwas anders, aber sie behalten drei gemeinsame Argumente bei, die wir hier erklären werden:

  • obj. Hier geben wir das Objekt an, mit dem die Funktion auf eine Kollision prüfen soll. Es kann alle eine Instanz - ID, ein Objekttyp, oder das besondere GM Schlüsselwort sein. Es ist auch erwähnenswert, dass, wenn der ausgewählte Objekttyp das designierte Elternobjekt anderer Objekte ist, alle untergeordneten Objekte dieses Objekts ebenfalls in diese Überprüfung einbezogen werden.


  • voraus. Dies kann entweder sein true oder false und bezieht sich auf die Genauigkeit der Kollisionsprüfung. Wenn auf eingestellt true Dann müssen präzise Kollisionen berechnet werden (aber nur dann, wenn das zu prüfende Objekt in seinen Maskeneigenschaften "präzise" für seine Kollisionserkennung markiert hat), was eine etwas komplexere Aufgabe für die CPU ist und daher mit Vorsicht verwendet werden sollte. Ob false Dann werden Kollisionen basierend auf der Begrenzungsbox des Objekts berechnet, für das geprüft werden soll.


  • Notme. Möglicherweise müssen Sie nach Kollisionen mit einem Objekt suchen, das denselben Index wie das Objekt hat, das den Code ausführt. Um zu verhindern, dass der Code eine Kollision mit sich selbst findet, können Sie dies auf Folgendes setzen true. Wenn Sie speziell nach der Instanz des Objekts suchen, auf dem der Code ausgeführt wird, oder nicht wichtig ist, kann sie als belassen werden false.

Alle diese Funktionen geben entweder die ID einer Instanz, die kollidiert, oder das spezielle Schlüsselwort noone zurück, wenn keine Kollision vorliegt. Beachten Sie, dass bei mehreren Kollisionen mit den Bereichen, die durch diese Funktionen und Instanzen des angegebenen Objekts definiert sind, nur eine Instanz-ID zurückgegeben wird und es sich um eine der Instanzen in der Kollision handeln kann.

Die folgenden Funktionen existieren, die sich mit erweiterten Kollisionen befassen.

  • collision_circle
  • Kollision_ellipse
  • Kollisionslinie
  • Kollisionspunkt
  • collision_rectangle

Alle oben genannten Funktionen sind auf Kollisionsprüfung Instanzen, und als solche stützen sich auf die Kollisionsmaske bezogen, die für die Instanz definiert ist. Es gibt jedoch viele Momente, in denen Sie nach "Kollisionen" mit einem Punkt oder einem Bereich suchen müssen, insbesondere wenn Ihrer Instanz kein Sprite zugewiesen ist oder wenn Sie mit der Maus usw. arbeiten. Daher auch GameMaker Studio 2 bietet die folgenden Funktionen, um Ihnen in diesen Situationen zu helfen:

  • Point_in_rectangle
  • Punkt_in_Dreieck
  • point_in_circle
  • rectangle_in_rectangle
  • rectangle_in_triangle
  • rectangle_in_circle

Bei Verwendung der integrierten Physik funktionieren die obigen Kollisionsfunktionen nicht für physikaktivierte Instanzen. Dies ist hauptsächlich auf die Tatsache zurückzuführen, dass diese Instanzen nicht mehr die Mehrheit der regulären eingebauten Variablen verwenden (stattdessen haben physikaktivierte Instanzen ihre eigenen Variablen ) und auch nicht die Kollisionsmaske oder die Begrenzungsbox, da sie Fixtures verwenden stattdessen. Es gibt jedoch eine spezielle Funktion zum Testen von Kollisionen mit Physik, die nach überlappenden Fixtures sucht:

  • physics_test_overlap
