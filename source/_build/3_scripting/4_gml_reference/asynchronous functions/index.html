Asynchrone Funktionen


Bevor Sie sich in GameMaker Studio 2 mit den asynchronen Funktionen (im Folgenden als Async- Funktionen bezeichnet) beschäftigen, fragen Sie sich vielleicht, was das Wort "asynchron" eigentlich bedeutet! Nun, die eigentliche Wörterbuchdefinition von asynchron ist:

bezieht sich auf eine Übertragungstechnik, die keinen gemeinsamen Takt zwischen den kommunizierenden Vorrichtungen erfordert; Taktsignale werden von speziellen Zeichen in dem Datenstrom selbst abgeleitet.

Im Kontext von GameMaker Studio 2 ist eine Async-Funktion eine Funktion, die an einen Webserver sendet oder nach Benutzereingaben fragt oder sogar Daten streamt, während GameMaker Studio 2 ohne Blockierung weiterläuft. Wenn der Informationsrückruf empfangen wird, wird ein spezielles Async-Ereignis ausgelöst, das wiederum jeder Instanz mit einem zugewiesenen Async-Ereignis erlaubt, weitere Funktionen auszuführen (obwohl dies nicht der Fall sein muss). Im Grunde ist es eine Möglichkeit, mit einer externen Ressource (Server, Benutzer oder Gerät) zu kommunizieren, ohne dass das Spiel oder das Projekt blockiert wird, während auf eine Antwort gewartet wird.

Warum ist das wichtig? Nun, da GameMaker Studio 2 beim Senden oder Empfangen von Daten weiterhin funktioniert, und das bedeutet, dass Sie viele Dinge tun können, ohne die Spielschleife zu blockieren -

  • streamen Sie Daten vom Gerät in Ihr Spiel, ohne dass der Spieler darauf wartet, dass Dinge geladen werden
  • Callback-Ereignisse haben, um Dinge nur zu tun, wenn die richtigen Informationen empfangen werden
  • kommunizieren und tauschen Daten mit einem Webserver aus

Dies alles macht alles viel flüssiger und unaufdringlicher. Es sollte beachtet werden, dass die asynchronen Ereignisse für alle Instanzen ausgelöst werden, in denen sie enthalten sind. Sie können also in einer Instanz eine asynchrone Funktion verwenden und das asynchrone Ereignis, das die Antwort behandelt, in einer anderen oder sogar in verschiedenen anderen Instanzen haben. Weitere Informationen zur Funktionsweise des asynchronen Ereignisses finden Sie hier - Asynchrone Ereignisse.

Die verschiedenen Arten von asynchronen Funktionen sind in den folgenden Abschnitten aufgeführt:

Im Folgenden finden Sie eine Liste der in GameMaker Studio 2 verfügbaren Async http-Funktionen:

  • http_request
  • http_get
  • http_get_file
  • http_post_string

Bitte beachten Sie, dass die oben genannten http_Funktionen möglicherweise aufgrund von domänenübergreifenden Sicherheitsproblemen nicht wie erwartet funktionieren. Dies bedeutet, dass Anforderungen an Ihren Server oder Versuche, Ressourcen aus verschiedenen Domänen zu laden, blockiert werden und möglicherweise leere Ergebnisse oder 404-Fehler zurückgeben. Eine Möglichkeit, dies zu umgehen, ist ein serverseitiges PHP, mit dem bestimmte Domänen auf Ihren Server zugreifen können (dies ist auch eine Möglichkeit, Ihre Ressourcen zu schützen und Server zu blockieren, die nicht in der PHP-Liste enthalten sind).

Im Folgenden finden Sie ein Beispiel für den Code, den Sie dafür verwenden können:

$http_origin = $_SERVER['HTTP_ORIGIN'];
if ($http_origin == "http://127.0.0.1:51268")
   {
   header('Access-Control-Allow-Origin: *');
   }


Frühere Versionen von GameMaker hatten Popup-Dialoge, die den Runner blockierten, während er auf eine Antwort des Benutzers wartete. Allerdings mögen die meisten modernen Geräte dies nicht und interpretieren den Läufer als Fehler und schließen die App. Um dies zu verhindern, verfügt GameMaker Studio 2 über asynchrone Entsprechungen dieser Funktionen, die den gleichen Zweck wie frühere Versionen erfüllen, aber den Spielrunner zu keiner Zeit blockieren, dh wenn das Spiel aufgerufen wird, läuft es normal im Hintergrund, während der Benutzer antwortet der gezeigte Dialog. Sobald der Benutzer geantwortet hat, wird ein asynchrones Ereignis für Dialoge aufgerufen, damit Sie mit den Ergebnissen umgehen können.

Im Folgenden finden Sie eine Liste aller verfügbaren asynchronen Dialogfunktionen:

  • get_login_async
  • show_message_async
  • show_question_async
  • get_string_async
  • get_integer_async

Beim Erstellen von Spielen ist es oft notwendig, Informationen über den Spielstatus in einer Datei eines bestimmten Typs zu speichern, aber das Speichern dieser Informationen auf dem Gerät ist nicht immer die beste Option, da der Spieler das Spiel löscht und dann neu installiert Informationen können verloren gehen. Um dies zu verhindern, können Sie verschiedene Cloud-Dienste nutzen, die jederzeit über das Internet Datenspeicher abrufen und ändern können. Sie sollten beachten, dass diese Funktion auf ein einzelnes Daten "Blob" pro Spiel beschränkt ist. Jedes Mal, wenn Sie eine neue Speicherung an den Cloud-Dienst senden, ob es eine Zeichenfolge oder eine Datei ist, werden zuvor gespeicherte Daten überschrieben.

GameMaker Studio 2 unterstützt dies mit einigen einfachen Funktionen, die in Verbindung mit dem Asynchronous Cloud Event funktionieren. Diese Funktionen werden auf den folgenden Seiten erläutert:

  • cloud_synchronise
  • cloud_string_save
  • cloud_file_save
HINWEIS: Zurzeit verwendet nur die Google-Dienste- Erweiterung für Android diese Funktionen. Diese Erweiterung muss vorhanden sein, damit Sie sie anrufen können.

Auf mobilen Geräten kann normalerweise immer nur eine Anwendung im Vordergrund aktiv sein, aber viele Spiele und Anwendungen arbeiten in einer zeitbasierten oder verbundenen Umgebung, in der Ereignisse von Interesse für Benutzer auftreten können, wenn die Anwendung nicht im Vordergrund steht. In diesen Fällen können lokale und Remote- Push-Benachrichtigungen diesen Spielen ermöglichen, ihre Benutzer zu benachrichtigen, wenn Ereignisse auftreten.

Wenn das Betriebssystem eine lokale oder Remote-Push-Benachrichtigung liefert und die Zielanwendung nicht im Vordergrund ausgeführt wird, wird dem Benutzer die Benachrichtigung in Form eines Banners angezeigt. Wenn es eine Benachrichtigungswarnung gibt und der Benutzer darauf tippt, wird das Spiel gestartet und ein asynchrones Push-Ereignis aufgerufen, in dem Sie die von Ihnen angegebene Rückrufzeichenfolge überprüfen und den Push basierend auf seinem Wert auflösen können. Wenn die Anwendung bei der Zustellung der Benachrichtigung im Vordergrund ausgeführt wird, erhält die Anwendung weiterhin ein asynchrones Push-Ereignis, der Benutzer erhält jedoch keine Meldung, es sei denn, Sie erstellen sie selbst.

Es gibt einige Unterschiede zwischen unterstützten Plattformen:

  • Unter Android werden alle Benachrichtigungen gespeichert, wenn die App nicht im Hintergrund ausgeführt wird, und die vom asynchronen Ereignis gelieferten Daten werden beim nächsten Start der App gespeichert, und Sie müssen Push-Benachrichtigungen innerhalb der Android Spieloptionen aktivieren Arbeit.

  • Unter iOS werden Daten für eine bestimmte Benachrichtigung nur übermittelt, wenn die Benachrichtigung ausgewählt ist (oder wenn die App im Vordergrund ausgeführt wird, wenn die Benachrichtigung empfangen wird). Sie müssen Push-Benachrichtigungen innerhalb der iOS-Spieloptionen aktivieren, damit sie funktionieren.

Eine lokale Push-Benachrichtigung ist lokal für das Gerät, auf dem das Spiel installiert ist, und erfordert keinen Back-End-Server. Sie legen einfach das Datum und die Uhrzeit für die Benachrichtigung fest und es wird dem Benutzer angezeigt, wenn das Spiel gerade nicht ausgeführt wird. Diese Art der Benachrichtigung ist nützlich, um "Erinnerungen" für den Benutzer zu setzen, um Ihr Spiel wieder zu spielen, oder um eine tägliche Belohnung für das Spielen anzubieten, usw. und wenn der Benutzer auf die Benachrichtigung tippt, wird es das Spiel starten und Daten eingeben Zeichenfolge, die dann vom Spiel analysiert werden kann, um eine Belohnung oder was auch immer zu geben.

Die folgenden Funktionen sind für lokale Benachrichtigungen verfügbar:

  • push_local_notification
  • push_get_first_local_notification
  • push_get_next_local_notification
  • push_cancel_local_notification
HINWEIS: Diese Funktionalität ist auf die Zielmodule für iOS und Android beschränkt. Für Android Sie die Google Play Services-Erweiterung installiert und Push-Benachrichtigungen in den Android Social Game-Optionen aktiviert haben.

Entfernte Benachrichtigungsnachrichten werden von einem Server an einen Dienst gesendet, der vom App Store der Geräteplattform bereitgestellt wird, und dieser leitet diese Nachrichten dann an alle Geräte weiter, auf denen Ihre Anwendung installiert ist. Dies wird von GameMaker Studio 2 auf den GameMaker Studio 2 für iOS und Android.

Es gibt keine Funktionen in GameMaker Studio 2, um mit Remote-Benachrichtigungen umzugehen, da sie alle von Ihrem Server generiert und von den jeweiligen App-Stores gehandhabt werden müssen. Sobald die Spiele korrekt eingerichtet sind, erhalten GameMaker Studio 2 Spiele diese Benachrichtigungen, die dann im asynchronen Push-Ereignis wie bei einer lokalen Benachrichtigung behandelt werden können.

Wenn das Spiel auf einem Gerät ausgeführt wird, wird es nach der notwendigen Einrichtung für alle verfügbaren Plattformen mit dem Plattform-Push-Benachrichtigungsdienst registriert. Dies wird ein asynchrones Push-Benachrichtigungsereignis auslösen, und das ds_map key " type " hat den Wert " register " sowie einen neuen Schlüssel " reg_id ", der die Registrierungs-ID enthält (oder eine Fehlermeldung in der " error " -Schlüssel, wenn " status " 0 ist). Sie müssen diese Registrierungs-ID dann an Ihren Server senden, und jedes Gerät, auf dem Ihr Spiel installiert ist, hat eine andere Registrierungs-ID. Ihr Server muss eine Liste von IDs für registrierte Geräte verwalten. Wenn Sie beispielsweise eine Push-Benachrichtigung von Ihrem Server senden, verwenden Sie die Registrierungs-IDs, um die Nachricht an die registrierten Geräte zu senden.

Bitte beachten Sie, dass es keine Garantie dafür gibt, dass Remote-Push-Benachrichtigungen zugestellt werden und die zulässige Datennutzlast relativ gering ist. Dies ist von Plattform zu Plattform unterschiedlich, aber iOS ist besonders eingeschränkt - der Apple-Dienst liefert nur die letzte Benachrichtigung, die vom Empfänger für die an Ihr asynchrones Ereignis zu liefernden Nutzdaten ausgewählt werden muss. Diese Benachrichtigungen haben eine maximale Nutzlast von 256 Byte. In der Regel würde eine Remote-Push-Benachrichtigung lediglich anzeigen, dass beispielsweise neue Daten von Ihrem Server verfügbar sind.

HINWEIS: Android erfordert, dass Sie die GCM Sender ID zu den Android Spieloptionen hinzufügen. Dies ist die Projektnummer, die beim Erstellen Ihres Google Play-API-Projekts zugewiesen wird.
