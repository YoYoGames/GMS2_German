Asynchrone Funktionen


Bevor mit den asynchronen Funktionen zu tun (bezeichnet von nun an als Asynchron - Funktionen) in GameMaker Studio 2 kann es sein, dass Sie sich fragen, was das Wort „asynchron“ eigentlich bedeutet! Nun, die eigentliche Wörterbuchdefinition von asynchron ist:

"eine Übertragungstechnik betrifft, die keinen gemeinsamen Takt zwischen den Kommunikationseinrichtungen erfordert; Zeitsignale werden von Sonderzeichen im Datenstrom selbst abgeleitet."

Im Kontext von GameMaker Studio 2 ist eine Async-Funktion eine Funktion, die an einen Webserver sendet oder nach Benutzereingaben oder sogar Streams von Daten fragt, während GameMaker Studio 2 ohne Blockierung weiter ausgeführt werden kann. Wenn der Informationsrückruf empfangen wird, wird ein spezielles Async-Ereignis ausgelöst, das wiederum jeder Instanz mit einem zugewiesenen Async-Ereignis die Ausführung weiterer Funktionen ermöglicht (obwohl dies nicht der Fall sein muss). Im Grunde ist dies eine Möglichkeit, mit einer externen Ressource (Server, Benutzer oder Gerät) zu kommunizieren, ohne dass das Spiel oder das Projekt blockiert werden, während auf eine Antwort gewartet wird.

Warum ist das wichtig? Nun, da GameMaker Studio 2 während des Sendens oder Empfangens von Daten weiterhin funktioniert, können Sie viele Dinge tun, ohne die Spielschleife zu blockieren.

  • Übertragen Sie Daten vom Gerät in Ihr Spiel, ohne dass der Spieler darauf wartet, dass etwas geladen wird
  • Callback-Ereignisse haben, um nur dann etwas zu tun, wenn die richtigen Informationen empfangen werden
  • Daten austauschen und Daten mit einem Webserver austauschen

Dadurch funktioniert alles viel fließender und unauffälliger. Es ist zu beachten, dass die asynchronen Ereignisse für alle Instanzen ausgelöst werden, über die sie verfügen. Sie können also eine asynchrone Funktion in einer Instanz verwenden und das asynchrone Ereignis verwenden, das sich mit der Antwort in einer anderen oder sogar verschiedenen anderen befasst. Hier finden Sie weitere Informationen zur Funktionsweise des asynchronen Ereignisses - Asynchronous Events.

Die verschiedenen Arten von asynchronen Funktionen sind in den folgenden Abschnitten aufgeführt:

Nachfolgend finden Sie eine Liste der in GameMaker Studio 2 verfügbaren Async-http-Funktionen:

  • http_anforderung
  • http_get
  • http_get_file
  • http_post_string

Beachten Sie, dass die obigen http_-Funktionen aufgrund von domänenübergreifenden Sicherheitsproblemen möglicherweise nicht wie erwartet funktionieren. Dies bedeutet, dass Anforderungen an Ihren Server oder Versuche, Ressourcen aus verschiedenen Domänen zu laden, blockiert werden und möglicherweise leere Ergebnisse oder 404-Fehler zurückgeben. Eine Möglichkeit, dies zu umgehen, besteht in serverseitigem PHP, mit dem bestimmte Domänen auf Ihren Server zugreifen können (dies ist auch eine Möglichkeit, Ihre Ressourcen zu schützen und Server zu blockieren, die nicht in der Liste der zulässigen PHP-Server enthalten sind).

Das folgende Beispiel zeigt den Code, den Sie dafür verwenden können:

$http_origin = $_SERVER['HTTP_ORIGIN'];
if ($http_origin == "http://127.0.0.1:51268")
   {
   header('Access-Control-Allow-Origin: *');
   }

Für Image-Ladeanforderungen, bei denen die Bestimmung oder Einstellung ihres Ursprungsorttyps wichtig ist, gibt es folgende Funktionen:

  • http_get_request_crossorigin
  • http_set_request_crossorigin

In den meisten Fällen sollten diese Funktionen nicht verwendet werden müssen, aber wenn das Spiel auf einem gesicherten Server gespeichert ist, auf dem für bestimmte Elemente der Zugriff auf die Basisauthentifizierung erforderlich ist und beim Laden Sicherheitsfehler auftreten, wird der Herkunftstyp für Bildanforderungen festgelegt "Verwendungsnachweise" können erforderlich sein (im Gegensatz zur Standardeinstellung "anonymous").


Frühere Versionen von GameMaker hatten Popup-Dialoge, die den Läufer blockierten, während er auf eine Antwort des Benutzers wartete. Die meisten modernen Geräte mögen dies jedoch nicht und interpretieren das Anhalten des Läufers als Fehler und schließen die App. Um dies zu verhindern, verfügt GameMaker Studio 2 über asynchrone Äquivalente dieser Funktionen, die den gleichen Zweck wie frühere Versionen erfüllen, den Game Runner jedoch zu keinem Zeitpunkt blockieren. Das bedeutet, dass das Spiel beim Aufruf wie gewohnt im Hintergrund ausgeführt wird, während der Benutzer darauf reagiert der gezeigte Dialog. Sobald der Benutzer geantwortet hat, wird ein asynchrones Dialog-Ereignis aufgerufen, damit Sie mit den Ergebnissen umgehen können.

Nachfolgend finden Sie eine Liste aller verfügbaren asynchronen Dialogfunktionen:

  • get_login_async
  • show_message_async
  • show_question_async
  • get_string_async
  • get_integer_async

Beim Erstellen von Spielen ist es häufig erforderlich, Informationen über den Spielstatus in einer Datei eines bestimmten Typs zu speichern. Das Speichern dieser Informationen auf dem Gerät ist jedoch nicht immer die beste Option, da der Spieler das Spiel gelöscht und diese dann erneut installiert hat Informationen können verloren gehen. Um dies zu verhindern, können Sie verschiedene Cloud-Dienste verwenden, die jederzeit Datenspeicherung über das Internet zum Abrufen und Ändern bieten. Sie sollten beachten, dass diese Funktion auf ein einzelnes "Daten-Blob" pro Spiel beschränkt ist. Jedes Mal, wenn Sie eine neue Sicherung an den Cloud-Dienst senden, wird dies, unabhängig davon, ob es sich um eine Zeichenfolge oder eine Datei handelt, alle zuvor gespeicherten Daten überschreiben.

GameMaker Studio 2 unterstützt dies mit ein paar einfachen Funktionen, die in Verbindung mit dem Asynchronous Cloud Event arbeiten. Diese Funktionen werden auf den folgenden Seiten erläutert:

  • cloud_synchronise
  • cloud_string_save
  • cloud_file_save
HINWEIS: Momentan verwendet nur das Android Ziel diese Funktionen, und Sie müssen das Kontrollkästchen Google Cloud Saving aktivieren im Social- Bereich der Android Spieloptionen aktivieren. Daraufhin werden Sie aufgefordert, die Google Services- Erweiterung herunterzuladen, die für das Cloud-Speichern auf Android erforderlich ist. Weitere Informationen zum Cloud Saving finden Sie im folgenden Helpdesk-Artikel:

Auf mobilen Geräten kann normalerweise immer nur eine Anwendung im Vordergrund aktiv sein. Viele Spiele und Anwendungen werden jedoch in einer zeitbasierten oder verbundenen Umgebung ausgeführt, in der für Benutzer interessante Ereignisse auftreten können, wenn sich die Anwendung nicht im Vordergrund befindet. In diesen Fällen können lokale und Remote- Push-Benachrichtigungen diese Spiele dazu veranlassen, ihre Benutzer zu benachrichtigen, wenn Ereignisse auftreten.

Wenn das Betriebssystem eine lokale oder Remote-Push-Benachrichtigung ausgibt und die Zielanwendung nicht im Vordergrund ausgeführt wird, wird dem Benutzer die Benachrichtigung in Form eines Banners angezeigt. Wenn eine Benachrichtigung angezeigt wird und der Benutzer darauf tippt, wird das Spiel gestartet und ein asynchrones Push-Ereignis aufgerufen, in dem Sie die von Ihnen angegebene Callback-Zeichenfolge überprüfen und den Push basierend auf seinem Wert auflösen können. Wenn die Anwendung bei Übermittlung der Benachrichtigung im Vordergrund ausgeführt wird, empfängt die Anwendung weiterhin ein asynchrones Push-Ereignis. Der Benutzer erhält jedoch keine Meldung, es sei denn, Sie erstellen es selbst.

Es gibt einige Unterschiede zwischen den unterstützten Plattformen:

  • Unter Android werden alle Benachrichtigungen, die empfangen werden, wenn die App nicht im Hintergrund ausgeführt wird, gespeichert und die Daten werden vom asynchronen Ereignis bei der nächsten Ausführung der App übermittelt. Sie müssen Push-Benachrichtigungen in den Android Spieloptionen aktivieren, damit sie Android Arbeit.

  • Unter iOS werden Daten für eine bestimmte Benachrichtigung nur geliefert, wenn die Benachrichtigung ausgewählt ist (oder wenn die App beim Empfang der Benachrichtigung im Vordergrund ausgeführt wird), und Sie müssen Push-Benachrichtigungen in den iOS-Spieloptionen aktivieren, damit sie funktionieren.

Eine lokale Push-Benachrichtigung ist lokal auf dem Gerät, auf dem das Spiel installiert ist, und erfordert keinen Back-End-Server. Sie legen einfach das Datum und die Uhrzeit für die Benachrichtigung fest, und es wird dem Benutzer angezeigt, wenn das Spiel derzeit nicht läuft. Diese Art der Benachrichtigung ist nützlich, um "Erinnerungen" für den Benutzer festzulegen, damit er Ihr Spiel erneut spielen kann, oder um eine tägliche Belohnung für das Spielen usw. anzubieten. Wenn der Benutzer die Benachrichtigung antippt, startet er das Spiel und leitet Daten ein Zeichenfolge, die dann vom Spiel analysiert werden kann, um eine Belohnung zu geben oder was auch immer.

Folgende Funktionen stehen für lokale Benachrichtigungen zur Verfügung:

  • push_local_notification
  • push_get_first_local_notification
  • push_get_next_local_notification
  • push_cancel_local_notification
HINWEIS: Diese Funktion ist auf die iOS- und Android Zielmodule beschränkt. Für Android Sie die Google Play Services-Erweiterung installiert haben und Push-Benachrichtigungen in den Android Social Game-Optionen aktiviert haben.

Remote-Benachrichtigungsnachrichten werden von einem Server an einen Dienst gesendet, der vom App Store der Geräteplattform bereitgestellt wird. Dieser leitet diese Nachrichten dann an alle Geräte weiter, auf denen Ihre Anwendung installiert ist. Dies wird von GameMaker Studio 2 auf den iOS- und Android Zielmodulen unterstützt.

In GameMaker Studio 2 gibt es keine Funktionen für Remote-Benachrichtigungen, da diese alle von Ihrem Server generiert und von den jeweiligen App Stores gehandhabt werden müssen. Nach der korrekten Einrichtung erhalten GameMaker Studio 2 Spiele diese Benachrichtigungen, die dann wie bei einer lokalen Benachrichtigung im Asynchronous Push Event behandelt werden können.

Wenn Sie das Spiel auf einem Gerät ausgeführt haben, registriert es bei allen verfügbaren Plattformen dieses Gerät beim Plattform-Push-Benachrichtigungsdienst, sobald Sie die erforderlichen Einstellungen vorgenommen haben. Dadurch wird ein asynchrones Push-Benachrichtigungsereignis ausgelöst ds_map Schlüssel " Typ " enthält den Wert " Register " sowie einen neuen Schlüssel " reg_id ", der die Registrierungs-ID enthält (oder eine Fehlernachricht im Schlüssel " error ", wenn " status " 0 ist). Sie müssen diese Registrierungs-ID dann an Ihren Server senden. Jedes Gerät, auf dem Ihr Spiel installiert ist, hat eine andere Registrierungs-ID. Ihr Server muss eine Liste mit IDs für registrierte Geräte verwalten. Wenn Sie beim Senden einer Push-Benachrichtigungsnachricht von Ihrem Server die Registrierungs-IDs verwenden, wird die Nachricht an die registrierten Geräte gesendet.

Bitte beachten Sie, dass es keine Garantie gibt, dass Remote-Push-Benachrichtigungen zugestellt werden und dass die zulässige Datennutzlast ziemlich gering ist. Dies ist von Plattform zu Plattform unterschiedlich, aber iOS ist besonders eingeschränkt. Der Apple-Dienst liefert nur die aktuellste Benachrichtigung, die vom Empfänger ausgewählt werden muss, damit die Nutzdaten an Ihr async-Ereignis übermittelt werden können. Diese Benachrichtigungen haben eine maximale Nutzlastgröße von 256 Byte. In der Regel zeigt eine Remote-Push-Benachrichtigung lediglich an, dass beispielsweise neue Daten von Ihrem Server verfügbar sind.

HINWEIS: Für Android erforderlich, dass Sie die GCM Sender ID in den Android Spieloptionen hinzufügen. Dies ist die Projektnummer, die zugewiesen wird, wenn Sie Ihr Google Play-API-Projekt erstellen.
